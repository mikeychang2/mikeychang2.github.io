Object-Oriented Programming: Classes vs. Modules

Key Differences & Purpose of Classes & Modules

November 16, 2014

Inheritance in Ruby allows you to do many nifty things with your code. It allows you to pass on code between different classes so that the subclass inherits methods, variables, and constants from its superclass, so that you don’t have to manually write the same lines of code for each object. However, Ruby doesn’t allow multiple inheritance. This means that your class will only have one superclass, rather than multiple parent classes.

To bypass this single-parent dilemma, Ruby does allow you to implement multiple inheritance through modules. You can access the methods within certain modules as mix-ins. This opens up many possibilities; however, the key is to know when to use a class and when to use a module.

Classes

Classes are constructs that are used to create instances of itself - called class instances or class objects or simply, objects. These instances take on the state and behavior of the class. You use the class to create many of these objects. You can create just a couple or even a million - and none of them will interfere with each other. Essentially, the class provides the code that is common to all potential instances. However, when you instantiate (create) an instance, you can provide more specific inputs. Basically, you add code that makes each unique to that particular instance.

Classes, then, serve as blueprints or templates for creating objects. Its behavior (methods, variables, and constants) are passed to these objects. If you aren’t planning on creating any instances of a class, don’t use class. A module might be a better bet.

Modules

Modules are a collection of methods and constants. You can think of them kind of like hashes. It operates as a way to map one thing to another. Just like a hash, where you use the key to retrieve the value, you cal module method and variable names on an object to retrieve that function. The key difference from classes is that modules serve to provide methods that you can use across multiple classes.

Modules cannot be instantiated. This means that you won’t ever have an object that mirrors the behavior of a module, like you would with a class. However, when your class includes a module, a proxy superclass is generated in order to provide access to all the module methods (this is where multiple inheritance comes into play). Not only are you inheriting from your superclass, you are now also “inheriting" from the module.
